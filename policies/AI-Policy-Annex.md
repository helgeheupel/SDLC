# AI Policy -- Operational Annex

This annex provides the operational procedures, checklists, and tools that implement the AI Policy (HH-AIMS-POL-001). Each section maps 1:1 to the corresponding AI Policy section, translating governance commitments into actionable day-to-day procedures for the Helge Heupel Group team.

---

## 1. Scope Operationalization

*Implements AI Policy Section 1: Purpose and Scope*

### 1.1 How to Determine if an AI System is In Scope

An AI system is in scope of the AIMS if it meets ANY of these criteria:

- [ ] The system uses machine learning, deep learning, or statistical inference
- [ ] The system generates, classifies, predicts, or recommends content or decisions
- [ ] The system processes natural language, images, audio, or code using AI techniques
- [ ] The system is marketed or described as "AI-powered" or "AI-assisted" to customers
- [ ] The system uses foundation models (LLMs, vision models) from any provider

**If uncertain:** Consult the CISO Agent. When in doubt, classify as in-scope.

## 3. Governance Principles Procedures

*Implements AI Policy Section 3: AI Governance Principles*

### 3.1 Human Oversight Checklist (Principle 1)

Before any AI output reaches a customer or production:

- [ ] AI output has been reviewed by a qualified human (CTO for architecture, CISO for security, Product Owner for product features)
- [ ] Reviewer has confirmed the output is factually accurate within their domain
- [ ] Reviewer has confirmed no prohibited practices are implicated
- [ ] Output is marked as AI-assisted where it will be visible to end users
- [ ] PR approval recorded (PR merge = approval evidence)

### 3.2 Transparency Implementation (Principle 2)

**For Verity VE/EAM Product AI:**
- All AI-generated or AI-assisted content displays a visual "AI-assisted" indicator
- Confidence scores are shown alongside AI recommendations (green >= 85%, amber 70-84%, red < 70%)
- "Why this recommendation?" panel is available for user-requested explanations
- Users can override, modify, or reject any AI recommendation

**For Claude Code Agent Outputs:**
- All agent outputs are produced in a tracked Git workflow
- Agent authorship is recorded in document metadata (author: CISO Agent, CTO Agent, etc.)
- PR description identifies AI-generated content for reviewer awareness

**For AI-Generated Communications:**
- Any external communication drafted by an AI agent must be reviewed and sent by a human
- Customer-facing documentation generated by AI must be reviewed for accuracy before publication

### 3.3 Accountability Records (Principle 3)

Every AI system in the AI System Registry must have:
- **Named human owner:** Accountable for system performance and compliance
- **RACI assignment:** Per RACI-Startup.md for governance activities
- **Risk acceptance record:** For any accepted residual risks, documented with `[CEO-SIGNOFF-REQUIRED]`

## 4. AI System Classification Procedures

*Implements AI Policy Section 4: AI System Classification*

### 4.1 EU AI Act Classification Decision Tree

Follow this decision tree for each AI system or feature:

```
START: Is the AI system used in any way listed in EU AI Act Art. 5?
  |
  YES --> STOP. This is a PROHIBITED practice. Do not develop or deploy.
  |       Report immediately to CISO Agent and CEO.
  |
  NO --> Does the AI system fall under an Annex III category?
    |
    YES --> Does the Art. 6(3) exception apply (no significant risk of harm)?
    |   |
    |   YES --> LIMITED-RISK. Apply Art. 50 transparency obligations.
    |   |
    |   NO --> HIGH-RISK. Full Art. 9-15 compliance required.
    |          Escalate to CEO for risk acceptance and conformity assessment planning.
    |
    NO --> Does the AI system interact with humans, generate synthetic content,
    |      or perform emotion recognition (Art. 50)?
    |
    YES --> LIMITED-RISK. Apply transparency obligations.
    |
    NO --> MINIMAL-RISK. Apply internal best practices.
```

### 4.2 SDLC Usage Tier Classification

| Factor | Tier 1 (Assistant) | Tier 2 (Collaborator) | Tier 3 (Decision Support) |
|---|---|---|---|
| Output reaches customers | No | No | Yes (with human review) |
| Affects production systems | No | No (draft only) | Yes (with G4 gate approval) |
| Processes PII | No | May process (with DPIA) | Yes (with DPIA + DPA) |
| Financial impact of errors | Low | Medium | High |
| Human review timing | Within 24h | Before acceptance | Real-time |

### 4.3 Reclassification Triggers

Reclassify an AI system when:
- Intended use changes (e.g., from internal tool to customer-facing feature)
- New data categories are processed (e.g., adding PII processing)
- Regulatory guidance changes the interpretation of risk categories
- An incident reveals higher risk than initially classified
- Annual classification review (aligned with AI Policy semi-annual review)

## 5. Prohibited Practices Checklist and Decision Tree

*Implements AI Policy Section 5: Prohibited Practices*

### 5.1 New AI Use Case Assessment Checklist

Before starting development of any new AI feature or deploying any new AI tool, complete this checklist:

**EU AI Act Art. 5 Screening:**

- [ ] Does the AI system evaluate or score people based on social behavior? --> If YES: **STOP -- Prohibited (social scoring)**
- [ ] Does the AI system target vulnerabilities of specific groups (age, disability, economic situation)? --> If YES: **STOP -- Prohibited (exploitation of vulnerabilities)**
- [ ] Does the AI system use real-time biometric identification in public spaces? --> If YES: **STOP -- Prohibited (real-time biometric ID)**
- [ ] Does the AI system infer emotions in workplace or educational settings? --> If YES: **STOP -- Prohibited (emotion recognition)**
- [ ] Does the AI system scrape facial images from the internet or CCTV? --> If YES: **STOP -- Prohibited (untargeted facial scraping)**
- [ ] Does the AI system predict criminal behavior based on profiling? --> If YES: **STOP -- Prohibited (predictive policing)**
- [ ] Does the AI system use subliminal or manipulative techniques? --> If YES: **STOP -- Prohibited (subliminal manipulation)**
- [ ] Does the AI system categorize people by biometric data for sensitive attributes? --> If YES: **STOP -- Prohibited (biometric categorization)**

**HH-Specific Prohibition Screening:**

- [ ] Will AI-generated output reach customers without human verification? --> If YES: **STOP -- Implement deterministic verification gate**
- [ ] Will the AI system access Secret-classified data? --> If YES: **STOP -- Obtain CEO written authorization first**
- [ ] Will the AI system make risk acceptance decisions? --> If YES: **STOP -- Redesign to require human risk acceptance**
- [ ] Will the AI system modify production systems without human approval? --> If YES: **STOP -- Implement G4 human approval gate**
- [ ] Will customer data be used for AI model training? --> If YES: **STOP -- Obtain customer consent + DPA + DPIA first**
- [ ] Is the AI tool on the approved toolchain list? --> If NO: **STOP -- Submit tool for assessment and approval**

### 5.2 Prohibited Practice Decision Tree

```
Is your AI use case flagged by ANY checkbox above?
  |
  YES --> STOP development/deployment immediately.
  |       Contact: CISO Agent (ciso@hh-group / claude --agent agents/ciso/CLAUDE.md)
  |       Contact: CEO Helge Heupel
  |       Document: What was found, which prohibition applies, what action was taken
  |
  NO --> Proceed with classification (Section 4) and standard development process.
```

### 5.3 Annual Prohibited Practices Review

Every 6 months (aligned with AI Policy review), the CISO Agent:
1. Reviews all AI systems in the inventory against the current prohibited practices list
2. Checks for new EU AI Act implementing acts that modify Art. 5
3. Assesses whether any HH-specific prohibitions need updating based on business changes
4. Documents the review outcome and presents at the Monthly Strategic meeting

## 7. AI System Lifecycle Procedures

*Implements AI Policy Section 7: AI System Lifecycle Requirements*

### 7.1 AI System Registration Process

When a new AI system is to be developed, deployed, or procured:

**Step 1: Identify**
- Determine if the system meets the AI system definition (Section 1.1 of this annex)
- If yes, assign a preliminary system owner

**Step 2: Classify**
- Complete the EU AI Act classification decision tree (Section 4.1)
- Determine the SDLC usage tier (Section 4.2)
- Document classification rationale

**Step 3: Screen for Prohibited Practices**
- Complete the prohibited practices checklist (Section 5.1)
- If any prohibition applies, STOP and escalate

**Step 4: Register**
- Add the system to the AI System Registry with:
  - System ID (format: AI-{category}-{seq}, e.g., AI-DEV-008)
  - Name, description, intended use
  - EU AI Act classification and SDLC tier
  - Data inputs and outputs description
  - Named human owner
  - Lifecycle status (Development, Deployed, Monitoring, Deprecated, Retired)
  - Applicable ISO 42001 Annex A controls

**Step 5: Assess Impact**
- For High-Risk systems: complete full AI Impact Assessment (Phase 5 process)
- For Limited-Risk systems: complete transparency requirements assessment
- For Minimal-Risk systems: confirm internal best practices apply

**Step 6: Approve**
- Limited-Risk and Minimal-Risk: CISO approves registration
- High-Risk: CEO approves registration and conformity assessment plan
- Prohibited: system must not be registered or developed

**Note:** The formal AI System Registry and its YAML schema are defined in Phase 5. Until then, this process applies using the current inventory (Section 1.2 of this annex).

### 7.2 AI System Change Management

When making changes to an existing AI system:

| Change Type | Action Required | Approval |
|---|---|---|
| Model retraining (same data distribution) | Update registry, re-run performance tests | CTO |
| Model retraining (new data distribution) | Reclassification check, bias re-assessment, DPIA update if PII | CISO + CTO |
| Architecture change | New ADR, reclassification check, security review | CTO + CEO for High/Critical |
| Intended use change | Full reclassification, new impact assessment if risk level changes | CEO |
| Data source addition | Data quality assessment, provenance documentation, DPA if customer data | CISO |
| Feature expansion to new markets/jurisdictions | Regulatory assessment for new jurisdiction | CISO + Legal |

### 8.2 AI Incident Reporting Procedures

**What constitutes an AI-specific incident:**

| Incident Type | Description | Severity | Reporting |
|---|---|---|---|
| **Bias detection** | AI system produces outcomes with disparate impact ratio < 0.8 across protected attributes | High | Weekly Technical meeting; remediation plan within 72h |
| **Hallucination with business impact** | AI system generates factually incorrect output that reaches a customer or affects a business decision | High/Critical | Immediate CISO + CEO notification; system suspension if Critical |
| **Unauthorized data processing** | AI system processes data outside its authorized scope or classification level | Critical | Immediate CEO notification; system suspension; NIS2/GDPR assessment |
| **Model compromise** | Evidence of adversarial attack, data poisoning, or model extraction | Critical | Immediate CEO + CISO notification; system suspension; NIS2 reporting (24h/72h) |
| **Prohibited practice detection** | AI system found to perform a prohibited practice | Critical | Immediate system suspension; CEO notification; corrective action within 24h |
| **Performance degradation** | AI model accuracy degrades > 5% from baseline | Medium | Weekly Technical review; revalidation triggered |
| **Transparency failure** | AI system fails to provide required transparency information to users | Medium | Corrective action within 30 days |

### 8.3 NIS2 AI Incident Reporting Integration

When an AI incident also meets NIS2 significant incident criteria:
1. CISO initiates NIS2 incident reporting in parallel with AI incident response
2. For GmbH: BSI notification within 24h (initial), 72h (detailed) via BSI portal in German
3. For B.V.: NCSC-NL notification within 24h (initial), 72h (detailed); formal notifications in Dutch
4. Dual-notification assessment: check if GDPR data breach notification to AP/BfDI is also required

## 10. Agent-Specific AI Policy Compliance Guide

This section provides specific guidance for how the 7 Claude Code agents comply with the AI Policy in daily operations.

### 10.1 Trust Level Protocol

All agents operate at Trust Level 2 (Supervised Autonomous):

| Permitted | Not Permitted |
|---|---|
| Draft policy documents | Approve or sign-off on policies |
| Draft risk assessments | Accept risk on behalf of HH Group |
| Produce code and configuration | Deploy to production without human approval |
| Analyze data within classification boundaries | Access Secret-classified data without CEO authorization |
| Spawn subagents per protocol | Spawn subagents exceeding 2 re-spawn limit |
| Flag dependencies and concerns | Make cross-domain decisions unilaterally |

### 10.2 PR Review Requirements for Agent Outputs

Every artifact produced by a Claude Code agent must pass through the PR review process:

1. **Agent creates** the artifact on a feature branch or within the current working context
2. **PR is created** with description identifying: agent name, task description, ISO clauses addressed, open questions
3. **Human reviewers** (per RACI matrix) review the content for accuracy, completeness, and compliance
4. **Approver** (typically CEO) merges the PR, constituting formal approval
5. **Audit trail:** The complete Git history (branch, commits, PR comments, merge) serves as the audit trail for ISO 27001 cl. 7.5 and ISO 42001 cl. 7.5

### 10.3 Agent Data Handling Rules

| Data Classification | Agent Access Rules |
|---|---|
| **Public** | All agents may process freely |
| **Internal** | All agents may process for authorized tasks |
| **Confidential** | Agents may process only when the task explicitly requires it; output must be marked Confidential |
| **Secret** | No agent may process without explicit written CEO authorization per interaction |

### 10.4 Agent Incident Reporting

When an agent encounters an AI-related anomaly during operation:

1. **Detect:** Agent identifies unexpected behavior (incorrect output, data access anomaly, unexpected model response)
2. **Flag:** Agent marks the anomaly in its output with `[AI-INCIDENT-FLAG]` and describes what was observed
3. **Escalate:** Human reviewer assesses the flag during PR review
4. **Report:** If the anomaly constitutes an AI incident (per Section 8.2 of this annex), the CISO Agent is notified and incident management procedures apply
