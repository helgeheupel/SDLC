**Document header:**
- Title: "Helge Heupel Group -- Startup Phase Governance Model"

**Section 1: Governance Philosophy at Startup**

The CTO/CISO expert assessment warns: "startups don't die from coordination overhead -- not from missing ISO clauses." The startup governance model optimizes for velocity while maintaining compliance. Key principles:
- 2 meetings replace 5+ governance bodies from SDLC-v2 S.13.3
- Risk-tiering is PRESERVED -- not all changes are equal
- Automated checks replace manual gates for Low/Medium risk
- Human judgment concentrated on High/Critical risk decisions
- Git PR workflow = audit trail (no additional approval systems needed)

**Section 2: Meeting Structure**

**Meeting 1: Weekly Technical Governance (30 minutes)**
- Frequency: Weekly
- Duration: 30 minutes (hard stop)
- Attendees: CEO, CTO, CISO, IT Pentester (required); Strategy Partner (optional); Finance SM (by invitation)
- Agent support: SDLC Lead Agent prepares agenda; specialist agents prepare materials as needed
- Purpose: Technical decisions, risk assessment, deployment approvals (High/Critical), incident review, security findings
- Standing agenda:
  1. Open incidents / security findings (5 min)
  2. High/Critical risk items requiring approval (10 min)
  3. Architecture decisions / ADRs for review (10 min)
  4. Upcoming deployments requiring sign-off (5 min)
- Decision authority: CTO for technical decisions; CISO for security decisions; CEO for risk acceptance
- Output: Decision log (maintained in Git as meeting-notes/weekly-technical/YYYY-MM-DD.md)

**Meeting 2: Monthly Strategic Governance (60 minutes)**
- Frequency: Monthly
- Duration: 60 minutes
- Attendees: CEO, Strategy Partner, CTO, CISO (required); Finance SM (required for financial items); IT Pentester (by invitation)
- Agent support: Product/Strategy Agent prepares strategic review; CISO Agent prepares compliance status
- Purpose: Strategic alignment, OKR review, compliance posture, risk register review, management review (ISO 27001 cl. 9.3 + ISO 42001 cl. 9.3)
- Standing agenda:
  1. Strategic progress / OKR review (15 min)
  2. Compliance posture and certification progress (15 min)
  3. Risk register review and treatment status (15 min)
  4. Resource allocation and budget review (10 min)
  5. Action items and decisions (5 min)
- Decision authority: CEO for strategic and policy decisions; sign-off on risk acceptance
- Output: Management review minutes (maintained in Git as meeting-notes/monthly-strategic/YYYY-MM.md)
- Note: This meeting satisfies ISO 27001 cl. 9.3 "Management review" and ISO 42001 cl. 9.3 when compliance is on the agenda

**Section 3: Risk-Tiered Gate Model**

Map the 6 SDLC-v2 gates (G0-G5) to the 2-meeting structure with risk tiering.

Reference: SDLC-v2 Section 13.1 defines 4 risk tiers (Low, Medium, High, Critical) and Section 13.2 defines gate criteria.

| Gate | Low Risk | Medium Risk | High Risk | Critical Risk |
|------|----------|-------------|-----------|---------------|
| G0: Strategic Alignment | Auto-approve if OKR-linked | Auto-approve if OKR-linked | Monthly Strategic meeting | Monthly Strategic meeting |
| G1: Requirements Readiness | PR approval (CTO/CISO) | PR approval (CTO/CISO) | Weekly Technical review | Weekly Technical review + CEO |
| G2: Development Completeness | CI/CD checks pass | CI/CD checks pass | Weekly Technical review | Weekly Technical review + CEO |
| G3: Quality and Compliance | Automated tests pass | Automated tests + security scan | Weekly Technical review | Weekly Technical review + CISO sign-off |
| G4: Release Readiness | Auto-deploy if G3 passed | PR approval (CTO) | Weekly Technical approval | Weekly Technical approval + CEO sign-off |
| G5: Post-Release Validation | Automated monitoring | Automated monitoring + 24h check | Weekly Technical review | Dedicated review in Weekly Technical |

Key design decisions:
- Low/Medium risk items flow through automated checks and PR approvals -- no meeting required
- High risk items require Weekly Technical meeting approval
- Critical risk items require Weekly Technical meeting approval PLUS CEO/CISO explicit sign-off
- All gate passages are recorded in Git (PR merge = gate passage evidence for audit)

**Section 4: Gate Criteria at Startup**

For each gate, list the simplified Startup criteria (referencing SDLC-v2 S.13.2 for the full Growth criteria):

**G0 - Strategic Alignment (Orient -> Requirements):**
- Startup: Feature/initiative is linked to an OKR or strategic hypothesis; threat landscape consideration for High/Critical risk
- Growth adds: Formal WSJF score, competitive analysis, strategic traceability

**G1 - Requirements Readiness (Requirements -> Development):**
- Startup: Requirements documented in PR/issue; threat model exists for High/Critical risk features; compliance requirements tagged
- Growth adds: Full user story schema, formal acceptance criteria, AI impact assessment

**G2 - Development Completeness (Development -> Testing):**
- Startup: Code passes linting and basic tests; PR reviewed by at least 1 human; security-sensitive code reviewed by CTO or CISO
- Growth adds: Full code coverage thresholds, architecture conformance checks, intent contract validation

**G3 - Quality and Compliance (Testing -> Deployment):**
- Startup: Unit/integration tests pass; SAST scan clean (no Critical/High findings); dependency vulnerability check clean; manual security review for High/Critical risk
- Growth adds: Full test suite, DAST, performance testing, AI model validation, compliance evidence package

**G4 - Release Readiness (Deployment -> Production):**
- Startup: Rollback plan documented; deployment runbook exists for High/Critical risk; monitoring/alerting configured
- Growth adds: Canary deployment, traffic management, full ORR checklist

**G5 - Post-Release Validation (Production -> Steady State):**
- Startup: No Critical incidents within 24h; basic monitoring confirms expected behavior; user-reported issues triaged
- Growth adds: SLO validation, error budget tracking, formal postmortem if SLO breach
