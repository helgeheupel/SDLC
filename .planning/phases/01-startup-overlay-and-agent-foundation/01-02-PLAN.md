**Create `protocols/subagent-protocol.md`:**

Document the subagent spawning protocol covering:

**1. Architecture Constraints:**
- Claude Code subagents CANNOT spawn other subagents (hard limitation)
- Maximum depth: 2 levels (parent agent + subagent)
- Subagents inherit parent's permission context but can have restricted tools
- Subagents run in their own context window; results return to parent
- This means: CEO -> SDLC Lead (session) -> specialist subagent (Task tool) is the deepest chain
- If a specialist needs sub-delegation, it must complete its task and produce a HANDOFF.md for the CEO to pass to another specialist

**2. When to Spawn:**
| Trigger | Example | Action |
|---------|---------|--------|
| Cross-domain task requiring specialist expertise | "Review threat model for AI pipeline" | SDLC Lead spawns CTO or AppSec subagent |
| Focused research within a domain | "What does SDLC-Growth-v2 say about NIS2 timelines?" | Any agent spawns haiku research subagent |
| Large artifact requiring parallel analysis | "Assess all 93 Annex A controls" | Agent spawns multiple focused subagents for control groups |
| Decision requiring specialist input | "Should we use Azure Key Vault or HashiCorp Vault?" | Agent spawns CTO subagent for analysis |

**When NOT to spawn:**
- Task is within the current agent's domain expertise
- Task is simple enough to complete in the current session
- Task would require the subagent to spawn its own subagents (depth violation)
- Task requires interactive human dialogue (use HANDOFF.md instead)

**3. Spawning Governance (token budget):**

| Trigger Type | Model | maxTurns | Tools | Budget Rationale |
|-------------|-------|----------|-------|------------------|
| Quick lookup / fact-check | haiku | 10 | Read, Grep, Glob | ~3K token prompt, fast, read-only |
| Focused analysis | sonnet | 25 | Read, Grep, Glob, Bash | ~10K token prompt, balanced capability/cost |
| Complex reasoning / drafting | opus | 50 | Read, Write, Edit, Bash, Grep, Glob | ~15K token prompt, maximum capability |
| Exploratory research | haiku | 15 | Read, Grep, Glob, WebFetch | ~3K token prompt, broad search |

Default: sonnet with 25 turns unless specific requirements dictate otherwise.

**4. Context Passing Protocol:**
- The spawn prompt (Task tool prompt) MUST include:
  a. Clear task description (what to produce)
  b. Relevant file paths to read (not the content itself -- let the subagent read)
  c. Constraints and boundaries (what NOT to do)
  d. Expected output format (what the parent expects back)
  e. SDLC-Growth-v2 section references if applicable
- Do NOT paste large documents into the spawn prompt. Instead, reference file paths.
- Do NOT include the entire parent agent's system prompt. Include only task-relevant context.

**5. Result Collection Protocol:**
- Subagent returns its result to the parent via the Task tool's return mechanism
- Parent validates the result against the task description
- If result is insufficient, parent can:
  a. Re-spawn with clarified instructions (preferred)
  b. Complete the work itself (if minor)
  c. Produce a HANDOFF.md for human review (if blocking)
- Subagent-produced files are immediately available to the parent (shared filesystem)

**6. Error Handling:**
- If subagent fails (tool errors, context overflow): parent completes the task itself or simplifies the request
- If subagent produces incorrect output: parent re-spawns with corrected instructions, never more than twice
- If subagent hits maxTurns limit: task was likely too large; parent breaks it into smaller sub-tasks
- If subagent attempts forbidden actions: log the attempt and complete manually

**7. Audit Trail:**
- Every subagent spawning should be traceable via the parent session's output
- For artifacts requiring sign-off: subagent creates file, parent creates PR
- The HANDOFF.md produced by any agent documents which subagents were used and for what
