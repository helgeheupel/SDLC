<objective>
Create the AI System Registry with full YAML entries for all 11 AI systems in AIMS scope, plus a consolidation script that generates an auditor-facing Markdown summary.

Purpose: The AI System Registry is the first evidence of operational AIMS scope per ISO 42001 Annex A.4.2. It formalizes what the AIMS Scope statements (Phase 2) identified, providing per-system documentation of intended use, risk classification, data flows, human oversight, transparency, and applicable controls. Auditors examine this registry to verify the organization understands each AI system's distinct purpose and risk profile.

Output: 11 per-system YAML files in ai-registry/, a Node.js consolidation script, and a generated AI-Registry-Summary.md.
</objective>

    Create the ai-registry/ directory and 11 YAML files using the 24-field schema from 05-RESEARCH.md Pattern 1. Each file uses YAML front-matter style (--- delimiters) consistent with Phase 3 risk register and SoA patterns.

    **ID convention:**
    - AI-AGENT-001 through AI-AGENT-007: 7 Claude Code agents (CISO, CTO, SDLC Lead, AI Science, Platform/SRE, Product/Strategy, AppSec)
    - AI-3PARTY-001: ChatGPT, AI-3PARTY-002: Codex
    - AI-PROD-001: Verity VE AI, AI-PROD-002: Verity EAM AI

    **Critical: Each entry MUST be system-specific (anti-pattern from research):**
    - Each agent has a DISTINCT `intended-use` reflecting its specific domain (security vs. architecture vs. testing vs. AI research vs. platform vs. product vs. application security)
    - Each agent has DIFFERENT `data-inputs` (CISO reads risk register and standards; CTO reads architecture docs and codebase; AppSec reads threat models and vulnerability reports; AI Science reads model evaluation data; Platform/SRE reads infrastructure configs; Product/Strategy reads market data and OKRs; SDLC Lead reads all agent outputs for orchestration)
    - Each agent has DIFFERENT `applicable-controls` reflecting its specific ISO 42001 Annex A relevance
    - Third-party tools (ChatGPT, Codex) have fewer applicable controls (primarily A.9 use and A.10 third-party)
    - Product AI (VE, EAM) has the MOST applicable controls because customer-facing AI requires A.5 impact, A.6 lifecycle, A.7 data, A.8 information controls
    - Verity product AI systems have `lifecycle-status: development` (not yet deployed to customers)
    - SDLC Lead, AI Science, Platform/SRE, and Product/Strategy agents have `lifecycle-status: development` (not yet created, placeholder agents from Phase 1)

    **Schema fields (24 total per entry):** system-id, name, category, description, provider, model, intended-use, operational-context, eu-ai-act-classification, eu-ai-act-rationale, sdlc-usage-tier, human-oversight, data-inputs (array of type/classification/description), data-outputs (array), transparency-information, lifecycle-status, deployment-entities, responsible-owner, applicable-controls (array of 42001-A.X.Y IDs), risk-references (array of RISK-* IDs), last-risk-assessment, last-impact-assessment, conformity-status, created-date, last-updated, notes.

    Use the complete example entries in 05-RESEARCH.md (Pattern 1 for AI-AGENT-001, Code Examples for AI-3PARTY-001 and AI-PROD-001) as templates but ensure every entry is unique and system-specific.

       - Section 1: Introduction (purpose, scope, methodology)
       - Section 2: Summary Statistics (total systems, by category, by EU AI Act classification, by lifecycle status)
       - Section 3: Development AI Systems table (system-id, name, EU AI Act class, SDLC tier, lifecycle, owner)
       - Section 4: Third-Party AI Systems table (same columns)
       - Section 5: Product AI Systems table (same columns)
       - Section 6: Control Applicability Matrix (system vs. applicable controls heat map in Markdown)
       - Section 7: Approval block (CEO sign-off, CISO review)
    6. Print summary stats to console

    **package.json:** name "ai-registry-scripts", version "1.0.0", dependency on js-yaml "^4.1.0"
