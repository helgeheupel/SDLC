    **Document 1: `policies/document-control.md`** (~200-300 lines)

    YAML frontmatter: document-id (HH-ISMS-DOC-001), title (Document Control Framework), version (v1.0), date (2026-02-19), classification (Internal), author (CISO Agent), reviewer ([CISO, CTO]), approver (CEO Helge Heupel), review-cycle (Annual), next-review (2027-02-19), status (Draft), change-summary (Initial release).

    Structure:

    1. **Purpose** -- Defines how documented information required by ISO 27001 and ISO 42001 is created, updated, controlled, and retained. Establishes Git as the single source of truth.

    2. **Scope** -- Applies to all documented information in the ISMS and AIMS: policies, procedures, work instructions, records, evidence, templates, scope statements, risk registers. Covers all 3 entities.

    3. **ISO 27001 cl. 7.5 Mapping to Git** -- This is the core section. Explicit mapping table (per research Pattern 5):

       | cl. 7.5 Requirement | Git Implementation | Evidence |
       |-----|-----|-----|
       | 7.5.1 General (required docs) | Document register (policies/document-register.md) | Register file in repo |
       | 7.5.2 Creating: identification | YAML frontmatter document-id field | Frontmatter in every document |
       | 7.5.2 Creating: format | Markdown (.md) with YAML frontmatter | Consistent format across all documents |
       | 7.5.2 Creating: review and approval | Pull Request with named reviewers + CEO merge | PR history in Git |
       | 7.5.3 Control: availability | Repository access for authorized personnel | Git repository permissions |
       | 7.5.3 Control: protection | Branch protection rules on main | GitHub/Azure DevOps branch policies |
       | 7.5.3 Control: distribution | Repository clone/access | Git access control |
       | 7.5.3 Control: storage and preservation | Git repository with cloud backup | Azure DevOps / GitHub storage |
       | 7.5.3 Control: change control | PR with diff review | Git commit + PR history |
       | 7.5.3 Control: retention | Git history (immutable), tag-based versions | Git log, tags |
       | 7.5.3 Control: disposition | Document status field (Active/Superseded/Archived) | Frontmatter status field |

    4. **Document Metadata Standard** -- Required YAML frontmatter fields for all controlled documents:
       - `document-id`: Unique ID (format: HH-{system}-{type}-{number}, e.g., HH-ISMS-POL-001)
       - `title`: Document title
       - `version`: Semantic version + date (v1.0, 2026-02-19 format)
       - `date`: ISO 8601 date of current version
       - `classification`: Public | Internal | Confidential | Secret
       - `author`: Who drafted (agent name or person)
       - `reviewer`: List of reviewers
       - `approver`: Who approved (always a human, typically CEO)
       - `review-cycle`: Annual | Semi-annual | Quarterly | As-needed
       - `next-review`: Date of next scheduled review
       - `status`: Draft | Active | Superseded | Archived
       - `change-summary`: Brief description of changes in this version
       - Optional: `parent-document`, `ims-context`, `entity`

    5. **Version Numbering Convention** -- Per locked decision: semantic + date stamp.
       - v1.0: Initial release
       - v1.1, v1.2: Minor updates (corrections, clarifications, procedural changes)
       - v2.0: Major revision (significant scope change, regulatory requirement change)
       - Each version tagged in Git: `doc/{document-id}/v{version}`

    6. **Approval Workflow** -- Per locked decision: CISO + CTO review, CEO approves.
       - Step 1: Author (CISO Agent or human) creates document on feature branch
       - Step 2: PR created with CISO and CTO as reviewers
       - Step 3: CISO and CTO review, comment, request changes
       - Step 4: Author addresses comments
       - Step 5: CEO approves and merges PR (this IS the approval signature)
       - Step 6: Document status updated to Active
       - Emergency changes: CEO can merge directly with post-hoc review at next Weekly Technical

    7. **Information Classification and Handling** -- 4 levels per locked decision:
       - **Public**: Published externally. No access restrictions. Example: marketing materials.
       - **Internal**: Available to all HH personnel and agents. Default classification. Example: policies, procedures.
       - **Confidential**: Need-to-know access. Example: risk register, audit findings, agent system prompts.
       - **Secret**: CEO + named individuals only. Example: patent-sensitive materials, financial projections.
       - Handling procedures per level: storage location, sharing restrictions, disposal requirements.
       - TISAX ISA alignment: map HH levels to TISAX protection need levels.

    8. **Document Review and Retention** -- Review triggers (scheduled, regulatory change, incident, organizational change). Retention periods: policies (permanent in Git history), records (7 years minimum per GDPR Art. 17 + tax law), superseded documents (archived, not deleted).

    9. **External Documents** -- How to handle documents of external origin (ISO standards, regulatory texts, vendor certifications). Store references, not copies. Track version/applicability.

    10. **Document Control for AI-Generated Content** -- Specific provisions for documents drafted by Claude Code agents. Must follow same approval workflow. Author field identifies agent + human reviewer. All AI-generated documents subject to human review before approval. Satisfies ISO 42001 human oversight requirement.
