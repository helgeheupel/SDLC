<objective>
Create a threat modeling framework with STRIDE/LINDDUN templates and produce two applied threat models (Claude Code agent system and Verity CMS) that validate the framework works for different system types.

Purpose: Threat models bridge the gap between the risk register (Phase 3) and the Annex A controls (SoA). They provide the risk-informed basis for SAST/DAST tool selection and configuration. ISO 27001 cl. 6.1.2 requires risk assessment to inform control selection; threat models make that connection explicit and auditable.

Output: Framework schema + templates, TM-CLAUDE-001.yml (Claude Code agent system), TM-VERITY-CMS-001.yml (Verity CMS with LINDDUN)
</objective>

  <action>
    Create the `security/threat-models/framework/` directory and populate it with three files.

    **threat-model-schema.yml:** Define the canonical YAML schema for all HH threat models. Use the schema from 04-RESEARCH.md Pattern 2 as the starting point, with these requirements:
    - YAML frontmatter block: threat-model-id (format: TM-[SYSTEM]-NNN), system-name, version, date, classification, author, reviewer, status, methodology (STRIDE|STRIDE+LINDDUN), linddun-applied (boolean)
    - system-description: 2-3 sentence description
    - data-flows: array of {id, name, source, destination, data-type, classification, trust-boundary-crossing}
    - trust-boundaries: array of {id, name, description, components-inside, components-outside}
    - components: array of {id, name, type (process|datastore|external-entity), technology, threats array}
    - Each threat: {threat-id, category (6 STRIDE categories), description, likelihood (1-5), impact (1-5), risk-score (calculated as likelihood x impact), mitigations array, residual-risk}
    - Each mitigation: {control-id (Annex A format A.X.Y), description, status (implemented|planned|not-started)}
    - risk-references: array of RISK-IDs from the risk register that this threat model relates to
    - linddun-threats section (only when linddun-applied: true): array of {threat-id, category (7 LINDDUN categories), component-id, description, gdpr-article, mitigations}
    - Likelihood and impact scales MUST match the 5x5 matrix in `risks/methodology/risk-assessment-methodology.md` for consistency

    **stride-template.yml:** A ready-to-use STRIDE-only template with placeholder values. Include comments explaining each field. Pre-populate the 6 STRIDE categories (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege) with example entries showing proper format.

    **linddun-template.yml:** A ready-to-use LINDDUN template extension (appended to STRIDE model when personal data is processed). Include the 7 LINDDUN categories (Linkability, Identifiability, Non-repudiation, Detectability, Disclosure of information, Unawareness, Non-compliance) with example entries. Include GDPR article references in each example.
  </action>

    4. STRIDE template has all 6 categories represented
    5. LINDDUN template has all 7 categories with GDPR article references
    6. Templates are valid YAML (parse without errors)
  </verify>

    **TM-CLAUDE-001.yml: Claude Code Agent System Threat Model**

    Apply STRIDE methodology to the 7-agent Claude Code system. This is an AI system under AIMS governance.

    System components to model:
    - CEO (external entity) -- human user interacting with Claude Code CLI
    - Claude Code CLI (process) -- local development tool executing agent prompts
    - Anthropic API (external entity) -- cloud API receiving prompts, returning responses
    - Agent CLAUDE.md files (datastore) -- system prompts stored in Git
    - Reference documents (datastore) -- domain-context.md, policies, risk register in Git
    - Git repository (datastore) -- version-controlled artifacts
    - Subagent spawning (process) -- Claude Code Task tool spawning child agents
    - Generated artifacts (datastore) -- policies, risk assessments, ADRs produced by agents

    Trust boundaries:
    - TB-001: Local machine boundary (CEO + CLI inside, Anthropic API outside)
    - TB-002: Git repository boundary (stored artifacts inside, runtime prompts outside)
    - TB-003: Agent privilege boundary (parent agent inside, subagent outside with restricted tools)

    Data flows to enumerate:
    - DF-001: CEO -> Claude Code CLI (prompt input)
    - DF-002: Claude Code CLI -> Anthropic API (prompt + context submission)
    - DF-003: Anthropic API -> Claude Code CLI (response)
    - DF-004: Claude Code CLI -> Git repository (artifact write)
    - DF-005: Git repository -> Claude Code CLI (context read)
    - DF-006: Parent agent -> Subagent (task delegation via Task tool)
    - DF-007: Subagent -> Parent agent (result return)

    For each component, enumerate STRIDE threats specific to this system (not generic). Example threats:
    - Spoofing: Unauthorized agent prompt modification changing agent behavior
    - Tampering: Malicious modification of reference documents (domain-context.md) altering agent outputs
    - Information Disclosure: Sensitive data (Confidential/Secret classification) included in prompts sent to Anthropic API
    - Denial of Service: Context window exhaustion via large reference file reads
    - Elevation of Privilege: Subagent exceeding tool permissions (e.g., Write tool when only Read granted)

    Each mitigation MUST reference an Annex A control (A.X.Y format).
    Include risk-references linking to relevant risk register entries (RISK-AI-*, RISK-SEC-*).
    LINDDUN does NOT apply (the agent system processes SDLC artifacts, not personal data per locked decision).

    **TM-VERITY-CMS-001.yml: Verity CMS Threat Model**

    Apply STRIDE + LINDDUN to the Verity CMS system. Model `verity-cms` (Payload CMS backend) per research recommendation, not Verity VE/EAM product logic.

    System components to model:
    - Admin users (external entity) -- content editors accessing CMS admin panel
    - Payload CMS admin (process) -- Next.js admin interface on Vercel
    - Payload CMS API (process) -- REST/GraphQL API for content operations
    - Neon PostgreSQL (datastore) -- managed database storing content
    - Vercel (external entity) -- hosting platform for CMS admin
    - Cloudflare Pages (external entity) -- hosting for Verity website consuming CMS content
    - Verity website (process) -- Qwik frontend consuming CMS API

    Trust boundaries:
    - TB-001: Internet boundary (admin users outside, CMS inside Vercel)
    - TB-002: Database boundary (CMS API inside, Neon PostgreSQL as managed service)
    - TB-003: CDN boundary (Vercel inside, Cloudflare Pages as separate deployment)

    Data flows: Admin -> CMS admin -> CMS API -> Neon DB, CMS API -> Verity website, external -> Verity website

    LINDDUN applies because the CMS may store personal data (content author information, potentially user data depending on use case). Include LINDDUN threats:
    - Linkability: CMS content metadata linking admin users across sessions
    - Identifiability: Admin user accounts identifiable via CMS admin panel
    - Non-compliance: GDPR Art. 17 right to erasure for admin user data in CMS

    Each mitigation MUST reference an Annex A control. Include risk-references linking to relevant risk register entries (RISK-SEC-*, RISK-OPS-*).
    Do NOT include patent-sensitive details about Verity VE/EAM -- reference as [PATENT-VE-001] if data flows touch product features.

    **Quality requirements for both models:**
    - Minimum 5 data flows per model
    - Minimum 3 trust boundaries per model
    - Minimum 4 components per model
    - Minimum 10 specific threats per model (not generic)
    - Every threat has at least 1 mitigation with Annex A control reference
    - At least 3 risk-references to risk register entries per model
    - Auditor-ready depth: an ISO 27001 auditor can follow the chain from risk register -> threat model -> control -> SoA -> evidence
