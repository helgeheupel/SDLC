    Sections:
    1. **Tool Selection**: Selected: `@cyclonedx/cyclonedx-node-pnpm` (official CycloneDX SBOM generator for pnpm projects). Rationale: HH repos use pnpm as package manager; dedicated pnpm generator reads `pnpm-lock.yaml` directly (not `package.json` ranges) for exact version resolution. Output: CycloneDX 1.5+ JSON format. Alternative considered: cdxgen (universal but less precise for pnpm-specific resolution). Alternative considered: `npm-sbom` (npm-specific, not pnpm).

    2. **Configuration**: Output format: CycloneDX JSON (`sbom.cdx.json`). Component types included: libraries, frameworks, applications. Metadata: tool version, timestamp, component tree with licenses. CRITICAL: SBOM must be generated AFTER `pnpm install --frozen-lockfile` to ensure lockfile integrity (see Pitfall 4 in research). SBOM must show exact versions (e.g., `3.2.1`), not ranges (e.g., `^3.0.0`).

    3. **Pipeline Integration**: GitHub Actions workflow trigger: on push to main (every merge generates updated SBOM), on release (published release gets SBOM attached as release artifact). SBOM uploaded as GitHub Actions artifact with `actions/upload-artifact@v4`. SBOM file naming: `sbom-{repo}-{git-sha}.cdx.json`.

    4. **Monthly Review Cycle (LOCKED DECISION)**: Monthly human review covering:
       - New dependencies added since last review
       - Dependencies removed
       - License changes (any dependency changing to copyleft license)
       - Known vulnerability status (new CVEs affecting current dependencies)
       - Dependency version freshness (major version behind = attention)
       - Review documented in `security/sbom/review-log.md` (created at first review, not in this plan)
       - Reviewer: CTO with CISO advisory
       - Review cadence: first week of each month

    5. **Reporting**: SBOM artifact stored as GitHub Actions artifact (90-day retention). Release SBOMs attached to GitHub releases permanently. Monthly review findings documented in review log. Evidence for audit: SBOM JSON file + review log entries.

    6. **Maintenance**: Tool version check: monthly. CycloneDX spec version alignment: when new CycloneDX versions release. SBOM accuracy validation: compare SBOM component count to `pnpm list --depth 0` output semi-annually.

    **HH-SENG-SCS-001.md: Supply Chain Security Framework**

    Create the supply chain security framework. Include YAML frontmatter (document-id: HH-SENG-SCS-001, classification: Internal, author: AppSec Agent, reviewer: CTO, CISO, status: Draft).

    This document is broader than a tool specification -- it defines the overall supply chain security strategy. Structure:

    1. **Supply Chain Security Strategy**: HH targets SLSA Level 2 for all production builds. SLSA Level 2 requires: (a) hosted build platform (GitHub Actions satisfies), (b) signed provenance (GitHub `actions/attest-build-provenance` satisfies). Scope: all production build artifacts from CI/CD. Exclusions: local development builds, preview deployments (preview provenance is aspirational, not mandatory).

    2. **SLSA Level Requirements and Evidence**:
       - SLSA Build Level 1: Build process defined (GitHub Actions workflows) -- SATISFIED
       - SLSA Build Level 2: Hosted build platform with signed provenance -- TARGET
       - SLSA Build Level 3: Hardened build platform (isolated builds) -- DEFERRED to Growth
       - Evidence: GitHub Actions workflow files + attestation records in Sigstore transparency log

    3. **Artifact Signing (Claude's Discretion -- Sigstore/Cosign per research recommendation)**:
       - Technology: Sigstore/Cosign keyless signing via OIDC
       - Rationale: Zero key management overhead, GitHub Actions OIDC identity, transparency log (Rekor), industry standard
       - What gets signed: SBOM files, build outputs pushed to main
       - Signing identity: GitHub Actions workflow identity (OIDC token)
       - Verification: `cosign verify-blob` with certificate identity matching GitHub Actions issuer
       - Alternative considered: GPG (requires manual key management, rotation, distribution -- not justified for Startup)

    4. **Provenance Generation**:
       - Tool: `actions/attest-build-provenance@v2` (GitHub native)
       - Format: in-toto attestation with Sigstore signing
       - Trigger: every push to main that produces build artifacts
       - Artifacts: build outputs (`.next/` for CMS, static output for websites)
       - Attestation storage: GitHub Attestation API (automatic via action)
       - Verification: `gh attestation verify` CLI command

    5. **Dependency Governance**:
       - New dependency review: required before adding any new production dependency
       - Review criteria: license compatibility (per HH-SENG-DEP-001.md whitelist), known vulnerabilities, maintenance status (last commit within 12 months), download volume (proxy for community trust)
       - SBOM cross-reference: every dependency in SBOM must be traceable to a `pnpm-lock.yaml` entry
       - Supply chain attack indicators: single-maintainer packages, recent ownership transfers, typosquatting name similarity

    6. **Provenance Verification Process**:
       - Pre-deployment: verify build provenance before promoting to production
       - SBOM: verify SBOM signature before using for compliance reporting
       - Procedure documented step-by-step for CTO and IT Pentester/Super Admin
       - Verification failures: block deployment, escalate to CTO + CISO

    7. **Annex A Control Mapping**:
       - A.5.21 (ICT Supply Chain): Supply chain security framework provides the control implementation
       - A.5.23 (Cloud Services): Provenance verification for cloud-deployed artifacts
       - A.8.25 (Secure Development Life Cycle): SLSA L2 as part of the SDLC
       - A.8.30 (Outsourced Development): Dependency governance for third-party code
       - Map each control to specific evidence artifact (framework doc, provenance attestation, SBOM)

    8. **Maintenance**: SLSA level assessment: annual review against SLSA specification updates. Sigstore/Cosign version updates: monthly. Provenance workflow review: quarterly. Dependency governance criteria: semi-annual review.

    **Reference config:**

    **provenance.yml**: Working GitHub Actions workflow based on 04-RESEARCH.md Example 3 and Example 4. Target: `helgeheupel-cms` repo. Include:
    - Trigger: push to main
    - Permissions: id-token write (Sigstore OIDC), contents read, attestations write
    - Steps:
      1. Checkout
      2. Setup pnpm + Node.js
      3. Install dependencies (frozen lockfile)
      4. Build (`pnpm build`)
      5. Generate SBOM (reuse pattern from sbom-cyclonedx.yml)
      6. Sign SBOM with Cosign: `cosign sign-blob --yes sbom.cdx.json --bundle sbom.cdx.json.bundle`
      7. Attest build provenance: `actions/attest-build-provenance@v2` with subject-path pointing to build output
      8. Upload signed SBOM and provenance bundle as artifacts
    - Comments explaining OIDC flow, transparency log, and verification commands
    - Note: this is a reference config; actual deployment combines SAST + SBOM + provenance into a single pipeline (documented in supply chain framework)
