    **HH-SENG-SAST-001.md: SAST Specification**

    Create the SAST specification following the 6-section pattern from 04-RESEARCH.md Pattern 3. Include YAML frontmatter (document-id: HH-SENG-SAST-001, classification: Internal, author: AppSec Agent, reviewer: CTO, CISO, status: Draft).

    Sections:
    1. **Tool Selection**: Selection criteria (open-source, TypeScript/JS support, GitHub integration, SARIF output, CI speed, custom rule support). Selected tool: Semgrep CE (LGPL-2.1). Rationale: free for all repos, native TS/JS/JSX/TSX support, 2000+ community rules, fast CI scans (~10s median), SARIF output for GitHub Security tab. Alternatives evaluated: CodeQL (requires GitHub Advanced Security for private repos), SonarQube Community (requires server infrastructure), ESLint security plugins (less comprehensive).

    2. **Configuration**: Base rule sets: `p/javascript`, `p/typescript`, `p/owasp-top-ten`. Severity mapping: Semgrep ERROR -> Critical/High (blocks), Semgrep WARNING -> Medium (advisory), Semgrep INFO -> Low (advisory). Custom rules: placeholder section for HH-specific patterns (e.g., patent data leak detection in Verity repos). Configuration file: `.semgrepconfig.yml` in each repo root.

    3. **Pipeline Integration**: GitHub Actions workflow trigger: on pull_request to main, on push to main. Scan runs in container (`semgrep/semgrep`). SARIF upload to GitHub Security tab via `github/codeql-action/upload-sarif@v3`. Separate severity check step blocks PR merge on Critical/High findings.

    4. **False Positive Management (Claude's Discretion -- centralized baseline approach per research recommendation)**: Use `.semgrepignore` for path-level exclusions (test fixtures, generated code, vendor dirs). Centralized `security/sast/sast-baseline.md` documenting all suppressed findings with date, reviewer, and justification. NO inline `# nosemgrep` comments (keep suppression decisions centralized and auditable). Quarterly review of baseline suppressions.

    5. **Reporting**: SARIF output format for GitHub integration. Weekly summary of new findings (if any). Monthly trend analysis as part of security review. Evidence format for audit: SARIF file + GitHub Security tab screenshot + baseline document.

    6. **Maintenance**: Semgrep version updates: monthly check for new releases. Rule set updates: automatic (community rules pull latest on each run). Custom rule review: semi-annual. Annual specification review and tool re-evaluation.

    **Annex A control references:** A.8.25 (SDLC), A.8.28 (Secure Coding), A.8.29 (Security Testing).

    Same 6-section structure. Include YAML frontmatter (document-id: HH-SENG-DAST-001).

    Sections:
    1. **Tool Selection**: Selected: OWASP ZAP (Apache 2.0). Rationale: most widely used open-source DAST, GitHub Actions support, API scanning, Docker-based CI integration. Alternatives: Nuclei (faster for known vulns but less comprehensive), Burp Suite Community (not CI-friendly).

    2. **Configuration**: Scan mode: baseline scan (quick, suitable for CI) on PRs; full scan on main branch pushes (scheduled weekly). Target URLs: Vercel preview deployments for PR scans, production URLs for scheduled scans. Authentication: configured for CMS admin panel testing (cookie-based auth). Scan policy: default OWASP ZAP policy with HH-specific context (technology stack: Next.js, Qwik, Payload CMS).

Report output in Markdown and HTML. Critical/High findings block PR (baseline scan). Full scan findings reported to security review.

    4. **False Positive Management**: ZAP context file for known false positive patterns. Alert filter configuration for technology-specific false positives (e.g., Next.js CSP headers, Vercel-specific headers). Documented in `security/dast/dast-baseline.md`.

    5. **Reporting**: HTML and Markdown reports per scan. GitHub check annotations. Monthly DAST summary in security review. Evidence for audit: scan reports + configuration files.

    6. **Maintenance**: ZAP version updates: monthly. Scan policy review: quarterly. Target URL updates: when deployment URLs change. Annual tool re-evaluation.

    **Annex A control references:** A.8.25 (SDLC), A.8.29 (Security Testing).

    **HH-SENG-DEP-001.md: Dependency Scanning Specification**

    Create the dependency scanning specification. Include YAML frontmatter (document-id: HH-SENG-DEP-001, classification: Internal, author: AppSec Agent, reviewer: CTO, CISO, status: Draft).

    Sections:
    1. **Tool Selection**: Primary: GitHub Dependabot (free, integrated). Rationale: built into GitHub, automated PRs for vulnerable dependencies, supports npm/pnpm ecosystems. Supplementary: `npm audit` / `pnpm audit` for ad-hoc scanning. Supplementary: CycloneDX SBOM license field analysis for license compliance (covered in Plan 04-05).

    2. **Configuration**: Dependabot.yml configuration for pnpm ecosystem. Update schedule: daily for security updates, weekly for version updates. Target branch: main. Assignees: CTO (for review). Labels: `dependencies`, `security`. Commit message prefix: `deps:`. Group minor/patch updates where safe. Open PR limit: 10 (prevent PR flood).

    3. **Pipeline Integration**: Dependabot creates PRs automatically. PR checks include SAST scan (from sast-semgrep.yml). PR merge criteria: Critical/High CVE fixes must be merged within SLA (Critical: 48h, High: 7 days, Medium: 30 days, Low: next release). Advisory-only for Medium/Low -- no pipeline blocking but tracked.

    4. **PR Blocking Criteria**: `npm audit --audit-level=critical` runs as part of CI. If Critical severity vulnerabilities exist in production dependencies, PR is blocked. High severity vulnerabilities generate warnings and require manual review. Medium/Low are logged but do not block.

    5. **License Compliance**: Allowed licenses: MIT, Apache-2.0, BSD-2-Clause, BSD-3-Clause, ISC, 0BSD, CC0-1.0. Warning licenses: LGPL-2.1, LGPL-3.0, MPL-2.0 (copyleft but permissive for library use). Blocked licenses: GPL-2.0, GPL-3.0, AGPL-3.0, SSPL, EUPL (strong copyleft, incompatible with proprietary distribution). License check: monthly via SBOM license field analysis. New dependencies with blocked licenses must not be merged.

    6. **Maintenance**: Dependabot configuration review: quarterly. SLA review: annually. License whitelist review: semi-annually. Audit log: track all Dependabot PRs merged in security review.

    **Annex A control references:** A.8.8 (Management of Technical Vulnerabilities), A.5.21 (ICT Supply Chain), A.8.30 (Outsourced Development).

    **Reference config:**

    **dependabot.yml**: Working Dependabot configuration for HH pnpm repos. Place at `.github/dependabot.yml`. Include:
    - version: 2
    - updates for npm ecosystem (pnpm uses npm ecosystem in Dependabot)
    - directory: "/" (root)
    - schedule: daily for security, weekly for version updates
    - reviewers: CTO
    - labels: dependencies, security
    - commit-message prefix: deps
    - open-pull-requests-limit: 10
    - groups: group minor and patch updates for non-security updates
    - Comments explaining each section
