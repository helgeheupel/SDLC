**Section structure (decision-tree pattern):**

1. **Purpose and Scope** — References IS Policy Section 6 (classification) and Annex 4 (access control procedures). Scope: all logical access to HH systems including cloud platforms, Git repositories, AI agent configurations, and development tools.

2. **Normative References** — Table: ISO 27001 Annex A controls A.5.15-A.5.18 and A.8.1-A.8.5, IS Policy, RACI matrix, governance model. Related Procedures table (HH-ISMS-PROC-001 for access-related incidents, HH-ISMS-PROC-006 for HR onboarding/offboarding access).

3. **Decision Tree: Access Lifecycle** — Three branches with clear flowchart-style logic:
   - 3.1 **New Access Request:** Who needs access? (Employee / Contractor / AI Agent) -> What system? -> What level? (Read / Write / Admin) -> Is it standard for the role? (Yes: auto-approve per RBAC. No: CTO approves. Admin: CEO + CTO approve) -> Provision access -> Log in access register.
   - 3.2 **Access Modification:** Reason? (Promotion / Role change / Project need / Temporary elevation) -> Is it an upgrade or downgrade? (Upgrade: approval required per level. Downgrade: auto-process. Temporary: JIT elevation with expiry) -> Modify access -> Update register.
   - 3.3 **Access Revocation:** Trigger? (Termination -> immediate revocation within 4 hours. Transfer -> adjust per new role. Contractor end -> revoke on contract end date. Security incident -> emergency revocation per HH-ISMS-PROC-001) -> Revoke access -> Update register -> Verify revocation.

4. **RBAC Model and Role Definitions** — Map the 6 HH human roles to standard access levels per system:
   - Table with rows: CEO, CTO, IT Pentester, Founders, Consultant (BearingPoint), External CISO (NorthGRC). Columns: GitHub (owner/maintainer/contributor/read), Vercel (admin/member/viewer), Cloudflare (admin/member), Neon (admin/reader), Azure (owner/contributor/reader), AI Agent Configs (write/read).
   - AI agent access: agents operate within Claude Code session scope, no persistent system credentials. Access controlled by session invocation and Trust Level 2 boundaries.

5. **Least Privilege and JIT Elevation** — Principle: all accounts start with minimum necessary access. JIT (Just-In-Time) elevation process:
   - Request elevation in team Signal group or PR comment stating reason and expected duration.
   - CTO approves for standard elevation. CEO + CTO for admin/privileged elevation.
   - Maximum elevation duration: 4 hours for admin access, 24 hours for elevated contributor access.
   - Auto-revert: manually tracked at Startup (calendar reminder). Growth target: automated JIT via Azure PIM.

6. **MFA Requirements** — All systems supporting MFA must have it enabled. Table of systems with MFA status:
   - GitHub: TOTP or hardware key (mandatory for all users)
   - Vercel: TOTP (mandatory for all users)
   - Cloudflare: TOTP (mandatory for all users)
   - Azure: Microsoft Authenticator or FIDO2 (mandatory for all users)
   - Neon: SSO via GitHub (inherits GitHub MFA)
   - Exception: AI agents do not authenticate via MFA (session-scoped, no persistent credentials).

7. **Quarterly Access Review Process** — Per locked decision:
   - **When:** First week of each quarter (January, April, July, October).
   - **Conductor:** IT Pentester generates access report per system.
   - **Review:** CTO reviews and approves standard access changes. CEO signs off on privileged/admin account access.
   - **Output:** Access review report stored in Git as `access-reviews/YYYY-QN-access-review.md`.
   - **Actions:** Remove stale accounts, downgrade unnecessary privileges, verify MFA compliance, update RBAC table if roles changed.
   - **Feeds into:** Management review (HH-ISMS-PROC-008).

8. **GDPR Data Subject Rights** — Per locked decision (GDPR embedded in access control):

   - Access to personal data: identify all systems containing subject's data using AI System Registry and system inventory.
   - Right to erasure: CTO coordinates technical deletion, CEO confirms completion.
   - Data portability: export in machine-readable format (JSON/CSV).

Create the decision-tree change management procedure as `procedures/change-management.md` with document ID HH-ISMS-PROC-004.

**YAML frontmatter:** Same pattern (document-id HH-ISMS-PROC-004, title "Change Management Procedure", date 2026-02-25, classification Internal, parent-document HH-ISMS-POL-001).

**Section structure (decision-tree pattern):**

1. **Purpose and Scope** — References governance model Section 3 (gate model). Scope: all changes to production systems, configurations, policies, and documentation. Includes infrastructure changes, code deployments, policy updates, and agent configuration changes.

2. **Normative References** — Table: ISO 27001 A.8.32 (Change management), A.5.37 (Documented operating procedures), governance model, RACI matrix. Related Procedures (HH-ISMS-PROC-001 for incident-triggered changes, HH-ISMS-PROC-002 for DR-triggered changes, HH-ISMS-PROC-003 for access changes).

3. **Change Classification** — Decision tree from governance model Section 3.2:
   - **Low risk:** Cosmetic changes, documentation updates, non-production changes. Criteria: no production impact, reversible, no security implications.
   - **Medium risk:** Standard feature deployments, dependency updates, staging config changes. Criteria: limited production impact, reversible within 1 hour.
   - **High risk:** Database schema changes, authentication changes, new external integrations, AI model updates. Criteria: significant production impact, complex rollback, security implications.
   - **Critical risk:** Infrastructure migrations, data migrations, security architecture changes, major version upgrades. Criteria: service disruption possible, irreversible or costly to reverse, regulatory implications.

4. **Approval Process** — Per locked decision:
   - **Low/Medium risk:** PR approval from any reviewer. Auto-merge after CI passes and 1 approval. No meeting required.
   - **High risk:** PR requires CEO + CTO approval. Discussed at next Weekly Technical meeting if time permits. Must include rollback plan in PR description.
   - **Critical risk:** PR requires CEO + CTO approval. Must be discussed at Weekly Technical meeting before merge. Requires documented rollback plan, testing evidence, and impact assessment in PR. CAB function exercised by CEO + CTO at the Weekly Technical.
   - Decision tree: New change -> classify risk level -> route to approval mechanism -> execute -> verify.

5. **Change Request Format** — All changes flow through Git PR. PR template must include:
   - Change description (what and why)
   - Risk classification (Low/Medium/High/Critical) with justification
   - Systems affected
   - Rollback plan (mandatory for High/Critical, recommended for Medium)
   - Testing evidence (screenshots, test output, or "manual verification" statement)
   - Security impact assessment (for High/Critical: does this change affect CIA?)
   Git PR = the change record. PR merge = change approval. Git history = audit trail. No separate change management system needed at Startup.

6. **Rollback Requirements** — Per risk tier:
   - **Low/Medium:** Git revert + redeploy. Rollback target: 15 minutes.
   - **High:** Documented rollback steps in PR. Tested rollback before merge if possible. Rollback target: 30 minutes.
   - **Critical:** Mandatory rollback rehearsal (dry-run in staging). Rollback steps verified by CTO before production execution. Rollback target: 1 hour.

7. **Emergency Change Process** — For incident-driven changes (referenced from HH-ISMS-PROC-001):
   - Emergency changes bypass normal approval: CTO can approve alone for SEV1/SEV2.
   - Must create PR within 24 hours after emergency change.
   - Post-emergency review at next Weekly Technical meeting.
   - Emergency changes logged in incident record.

8. **AI Agent Configuration Changes** — Changes to agent CLAUDE.md files, reference documents, or routing rules:
   - Low risk: Documentation updates, reference data refresh.
   - Medium risk: Prompt modifications, tool permission changes.
   - High risk: New agent creation, agent Trust Level changes, boundary modifications.
   - All agent changes require CEO approval per Trust Level 2 governance.

9. **Traceability** — Complete audit trail through Git:
   - PR number = change request ID
   - PR description = change request details
   - PR reviews = approval records
   - PR merge commit = change execution record
   - Deployment log (Vercel/Cloudflare) = implementation evidence
   - No additional traceability system at Startup. Growth target: Linear integration (Phase 9).

10. **Annex A Control Mapping** — A.8.32 (Change management) mapped to Sections 3-7. A.5.37 (Documented operating procedures) mapped to Section 5.

11. **Regulatory Obligations** — ISO 27001 (A.8.32, A.5.37), TISAX (ISA Chapter 5), NIS2 (informational — change management requirements).

12. **Document Control** — Version history and approval.

**CRITICAL:** The approval matrix MUST match the locked decision exactly: Low/Medium via PR approval (any reviewer), High/Critical require CEO + CTO approval on the PR.
