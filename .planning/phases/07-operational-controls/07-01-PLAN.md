Review stored in Git as `incidents/PIR-{YYYY-MM-DD}-{short-description}.md`. Feeds into management review (HH-ISMS-PROC-008) and risk register updates.

**Section structure (full runbook pattern):**

1. **Purpose and Scope** — References IS Policy Section 7. Scope: all services and systems identified in ISMS scope statements across all 3 entities. States this plan covers business continuity (maintaining operations during disruption) and disaster recovery (restoring systems after failure).

2. **Normative References** — Table: ISO 22301 (BCP), ISO 27031 (ICT readiness), ISO 27001 Annex A controls (A.5.29, A.5.30), TISAX ISA Chapter 10. Related Procedures table (HH-ISMS-PROC-001 for incident escalation to DR, HH-ISMS-PROC-004 for post-DR change management).

3. **Service Tier Classification and RPO/RTO Targets** — Extract from Platform/SRE domain context Section 5 (SLO framework). Create table:
   - **Tier 1 (Mission-Critical):** RPO 1h, RTO 4h. Services: Verity CMS production, customer-facing APIs.
   - **Tier 1-AI:** RPO 1h, RTO 4h. Services: AI agent endpoints, Azure OpenAI Service integration.
   - **Tier 2 (Business-Important):** RPO 4h, RTO 8h. Services: HH Admin CMS, internal tools, Git repositories.
   - **Tier 3 (Supporting):** RPO 24h, RTO 24h. Services: Development environments, staging, monitoring dashboards.
   Note: These are current Startup targets. Growth targets (shorter RPO/RTO) documented as callouts.

4. **Business Impact Analysis Summary** — For each service tier: what happens if unavailable for 1h, 4h, 24h, 72h. Financial impact (revenue loss for customer-facing), reputational impact, regulatory impact (NIS2 reporting triggers at 24h for significant incidents). Reference risk register entries RISK-OPS-001 through RISK-OPS-005.

5. **Current Infrastructure and Recovery Capabilities** — Honest Startup state:
   - **Neon PostgreSQL:** Automated point-in-time recovery (PITR), 7-day backup retention. Recovery: restore from Neon console to specific timestamp. RPO capability: ~minutes (continuous WAL streaming).
   - **Vercel:** Immutable deployments, instant rollback to any previous deployment via dashboard or CLI. Recovery: `vercel rollback` or promote previous deployment. RPO capability: zero (deployment artifacts preserved indefinitely).
   - **Cloudflare Pages:** Similar to Vercel — immutable builds with instant rollback. Cloudflare DNS propagation: minutes.
   - **Azure OpenAI Service:** Microsoft-managed, SLA 99.9%. Recovery: failover to different Azure region (manual reconfiguration). RPO: not applicable (stateless API).
   - **Git (GitHub):** Distributed by nature. Full clone exists on every developer machine and CI environment. Recovery: re-clone from GitHub or any developer machine.
   - **No IaC:** All provisioning is manual at Startup. Growth target: Terraform/Bicep with automated recovery.

6. **Disaster Recovery Procedures** — Step-by-step runbooks for each failure scenario:
   - 6.1 **Database failure (Neon):** (1) Confirm failure via Neon dashboard status. (2) Check Neon status page. (3) If Neon-side issue: wait for resolution, monitor. (4) If data corruption: initiate PITR restore to last known-good timestamp. (5) Verify data integrity post-restore. (6) Update DNS/connection strings if necessary. (7) Verification: run application health checks.
   - 6.2 **Hosting platform failure (Vercel/Cloudflare):** (1) Confirm via status page. (2) If deployment-specific: rollback to previous deployment. (3) If platform-wide: activate holding page (static HTML on alternative CDN). (4) Growth target: automatic failover to Azure Static Web Apps.
   - 6.3 **AI service failure (Azure OpenAI):** (1) Confirm via Azure status page. (2) AI agents degrade gracefully (manual fallback per SDLC-Growth-v2 Section 11.5). (3) If prolonged: disable AI-dependent features, operate manually. (4) No automatic failover at Startup.
   - 6.4 **Full cloud region failure:** (1) Extremely unlikely for multi-region providers (Vercel, Cloudflare, Neon). (2) If Azure West Europe fails: AI services unavailable. (3) Manual failover to Azure Germany West Central per ISMS scope. (4) Growth target: automated region failover.
   - 6.5 **Supply chain compromise (dependency poisoning):** Cross-reference HH-ISMS-PROC-005 (Supplier Security). Immediate: lock dependencies, audit SBOM, rollback if necessary.

7. **Communication During DR Events** — Same ICS-Lite structure as incident management. CEO = Communications Lead. Customer communication templates: service disruption notice, status update, resolution notice. Internal: CTO coordinates technical recovery, posts updates in team Signal group.

8. **Testing Schedule** — Per locked decision and research:
   - **Quarterly:** Tabletop exercise (walk through a DR scenario with all key personnel, no actual failover). First tabletop before Stage 1 audit.
   - **Semi-annually:** Neon PITR restore test (restore a backup to a test database, verify data integrity). Vercel rollback test (deploy, rollback, verify).
   - **Annually:** Full DR simulation (simulate a major scenario, execute recovery procedures, measure actual RPO/RTO against targets).
   - Document results in `bcp-dr/test-results/YYYY-MM-DD-{type}.md`. Feed into management review (HH-ISMS-PROC-008).

9. **Roles and Responsibilities** — Table: CTO (DR Coordinator, leads technical recovery), IT Pentester (assists recovery, validates security post-restore), CEO (Communications Lead, business decisions), CISO Agent (assists with compliance impact assessment).

10. **Annex A Control Mapping** — A.5.29 (ICT readiness for business continuity) mapped to Sections 3-6. A.5.30 (ICT readiness for business continuity) mapped to Section 8.

11. **Regulatory Obligations** — ISO 27001 (A.5.29-30), ISO 22301, TISAX (ISA Chapter 10), NIS2 (informational — BCP requirements in Growth).

12. **Document Control** — Version history and approval.

**CRITICAL anti-patterns:**
- Do NOT reference Kubernetes, canary deployments, Terraform, or other Growth-phase infrastructure as current
- RPO/RTO targets must be realistic for CURRENT infrastructure (Neon PITR + Vercel immutable deployments)
- Failover procedures are MANUAL at Startup — no automated failover except what providers offer natively
