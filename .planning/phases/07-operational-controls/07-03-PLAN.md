Create the decision-tree supplier security procedure as `procedures/supplier-security.md` with document ID HH-ISMS-PROC-005.

**YAML frontmatter:** Same pattern (document-id HH-ISMS-PROC-005, title "Supplier Security Procedure", date 2026-02-25, classification Internal, parent-document HH-ISMS-POL-001).

**Section structure (decision-tree pattern):**

1. **Purpose and Scope** — References IS Policy Section 5 (risk management) and AI Policy Section 4 (AI system classification). Scope: all third-party suppliers with access to HH information, systems, or data including cloud infrastructure providers, SaaS tools, consultants, and AI service providers.

2. **Normative References** — Table: ISO 27001 Annex A controls A.5.19-A.5.23, ISO 42001 Annex A.10 (third-party AI assessment), IS Policy, AI Policy, risk register (RISK-SUP-001). Related Procedures (HH-ISMS-PROC-003 for supplier access management, HH-ISMS-PROC-001 for supplier-triggered incidents).

3. **Decision Tree: Supplier Lifecycle** — Three branches:
   - 3.1 **New Supplier Onboarding:** Type of supplier? (Cloud/SaaS / Consultant / AI service / Other) -> What data will they access? (Public only -> minimal assessment. Internal -> standard assessment. Confidential/Secret -> enhanced assessment) -> Complete risk assessment -> Contractual clauses -> Provision access per HH-ISMS-PROC-003 -> Add to supplier register.
   - 3.2 **Ongoing Supplier Monitoring:** Annual review cycle. Check: security certifications still valid? -> service performance acceptable? -> any incidents? -> data processing changes? -> If material change -> re-assess.
   - 3.3 **Supplier Offboarding:** Revoke all access per HH-ISMS-PROC-003 -> Confirm data return/deletion -> Update supplier register -> Update risk register if residual risk changes.

4. **Vendor Risk Assessment** — Tiered assessment based on data classification:
   - **Tier 1 (Confidential/Secret data):** Full security questionnaire, ISO 27001 certification or equivalent required, penetration test results, DPA review by CEO. Suppliers: Azure (cloud infrastructure), Neon (database with customer data).
   - **Tier 2 (Internal data):** Security questionnaire (abbreviated), privacy policy review, DPA if personal data. Suppliers: Vercel (hosting), Cloudflare (CDN/DNS), GitHub (code repository).
   - **Tier 3 (Public data only):** Minimal assessment — privacy policy review, basic due diligence. Suppliers: domain registrars, analytics tools.
   - **Consultant assessment:** BearingPoint, NorthGRC — NDA required, access scoped to project needs, security awareness confirmation.

5. **Contractual Security Clauses** — Standard clauses to include in all supplier contracts:
   - Confidentiality / NDA (mandatory for all)
   - Data processing agreement (DPA) if personal data per GDPR Art. 28
   - Security incident notification (supplier must notify HH within 24 hours of breach)
   - Right to audit (for Tier 1 suppliers)
   - Sub-processor notification (for cloud providers)
   - Data location requirements (EU preferred per ISMS scope)
   - Termination data handling (return or certified destruction)

6. **GDPR Data Processing Agreements** — Per locked decision (GDPR DPA embedded in supplier security):
   - DPA required when supplier processes personal data on behalf of HH (processor relationship per GDPR Art. 28).
   - DPA checklist: processing purpose, categories of data, data subjects, duration, sub-processor list, international transfer mechanisms (SCCs if outside EU), technical and organizational measures.
   - Current DPA inventory: Azure (Microsoft DPA), Neon (DPA via ToS), Vercel (DPA via ToS), Cloudflare (DPA via ToS).
   - CEO as data controller signs all DPAs.

7. **Third-Party AI Assessment** — Per ISO 42001 Annex A.10:
   - All third-party AI services must be assessed before deployment.
   - Assessment criteria: intended use alignment with AI Policy, data handling practices, model transparency (can we explain outputs?), EU AI Act classification, human oversight measures.
   - Current third-party AI: ChatGPT (AI-3PARTY-001) — assessed, Limited-Risk. Codex (AI-3PARTY-002) — assessed, Limited-Risk.
   - New AI tool onboarding: complete AI risk assessment (HH-AIMS-RMA-001) before deployment. Register in AI System Registry.

8. **Supplier Register** — Maintained as `suppliers/supplier-register.md` (or YAML). Fields: supplier name, service provided, data classification accessed, risk tier, DPA status, last assessment date, next review date, contract expiry, primary contact.

9. **Annex A Control Mapping** — Table: A.5.19 (Information security in supplier relationships), A.5.20 (Addressing information security within supplier agreements), A.5.21 (Managing information security in the ICT supply chain), A.5.22 (Monitoring, review, and change management of supplier services), A.5.23 (Information security for use of cloud services). Map each to specific sections.

10. **Regulatory Obligations** — ISO 27001 (A.5.19-23), ISO 42001 (A.10), TISAX (ISA Chapter 9), GDPR (Art. 28 processor requirements), NIS2 (informational — supply chain requirements).

11. **Document Control** — Version history and approval.

<task type="auto">
  <name>Task 2: Create HR Security Procedure (HH-ISMS-PROC-006)</name>
  <files>procedures/hr-security.md</files>
  <action>
Create the decision-tree HR security procedure as `procedures/hr-security.md` with document ID HH-ISMS-PROC-006.

**YAML frontmatter:** Same pattern (document-id HH-ISMS-PROC-006, title "HR Security Procedure", date 2026-02-25, classification Internal, parent-document HH-ISMS-POL-001).

**Section structure (decision-tree pattern):**

1. **Purpose and Scope** — References IS Policy Section 4 (roles and responsibilities) and IS Policy Annex 8 (onboarding/offboarding). Scope: all personnel (employees, contractors, consultants) across GmbH, B.V., and Inc. entities.

2. **Normative References** — Table: ISO 27001 Annex A controls A.6.1-A.6.8, IS Policy, RACI matrix. Related Procedures (HH-ISMS-PROC-003 for access provisioning/revocation, HH-ISMS-PROC-005 for consultant/contractor engagement).

3. **Decision Tree: Personnel Security Lifecycle** — Three branches:
   - 3.1 **Pre-Employment / Onboarding:** Type? (Employee / Contractor / Consultant / AI Agent operator) -> Background check appropriate to role and jurisdiction (GmbH: Fuehrungszeugnis if applicable. B.V.: VOG if applicable. Inc.: standard background check) -> NDA signed -> IS Policy acknowledgment signed -> Security awareness briefing completed -> Access provisioned per HH-ISMS-PROC-003 -> Added to personnel register.
   - 3.2 **During Employment:** Annual security awareness refresher -> Role change? (Update access per HH-ISMS-PROC-003, update RBAC) -> Performance with security responsibilities assessed in regular reviews.
   - 3.3 **Termination / Transfer:** Reason? (Voluntary / Involuntary / Contract end / Transfer) -> If involuntary or security concern: immediate access revocation (within 4 hours per IS Policy Annex 8) -> Return all company assets (devices, access tokens) -> Confirm NDA survives termination -> Exit interview includes security reminder -> Update personnel register.

4. **Pre-Employment Screening** — Decision tree by jurisdiction and role:
   - **All roles:** Identity verification, employment history verification (prior 3 years), reference check (minimum 1).
   - **GmbH (Germany):** Criminal record check (polizeiliches Fuehrungszeugnis) for roles with Confidential/Secret data access. Note: German labor law limits pre-employment checks — consult legal if in doubt.
   - **B.V. (Netherlands):** Certificate of Good Conduct (Verklaring Omtrent het Gedrag — VOG) for roles with Confidential/Secret data access.
   - **Consultants/Contractors:** Company-level due diligence via HH-ISMS-PROC-005 (supplier security). Individual NDA signed.
   - CEO approves all personnel hiring decisions.

5. **Security Awareness Training** — Program structure:
   - **Initial:** Upon onboarding, within first 5 business days. Topics: IS Policy overview, classification scheme, acceptable use, incident reporting, GDPR basics, AI Policy overview (prohibited practices, Trust Level 2). Format: self-study of IS Policy + AI Policy + acknowledgment form.
   - **Annual refresher:** Topics: IS Policy updates, recent incidents/lessons learned, emerging threats relevant to HH, GDPR refresher, AI governance updates. Format: 30-minute briefing at Monthly Strategic meeting (Q1 meeting).
   - **Role-specific:** CTO/IT Pentester: technical security training (OWASP, secure coding). CEO: management review and regulatory update training.
   - **Evidence:** Training completion records stored in Git as `hr-security/training/YYYY-{name}-completion.md`.

6. **NDA Management** — Decision tree:
   - **Employee NDA:** Signed at hiring. Covers employment period + 2 years post-termination. Standard HH NDA template. CEO signs on behalf of HH entity.
   - **Consultant/Contractor NDA:** Signed at engagement per HH-ISMS-PROC-005. Duration matches contract + 2 years. May use mutual NDA if consultant has own template.
   - **AI Agent NDA:** Not applicable — AI agents do not sign NDAs. Agent confidentiality enforced via Trust Level 2 boundaries and prohibited practices in AI Policy.
   - **NDA register:** Maintained as part of HR records. Fields: name, entity, NDA date, duration, post-termination period, status (active/expired).

7. **Termination Procedures** — Detailed process per IS Policy Annex 8:
   - **Standard departure:**
     - Last day: access revoked per HH-ISMS-PROC-003 (all systems within 4 hours of departure)
     - Return company equipment (if any — cloud-first company, may be minimal)
     - Confirm NDA obligations with departing individual
     - Exit interview: security-focused questions (data handling, any concerns)
   - **Involuntary/security-related departure:**
     - Immediate access revocation (during or immediately after notification meeting)
     - CTO executes emergency access revocation per HH-ISMS-PROC-003
     - CEO communicates as needed to team
     - Post-departure: monitor for unauthorized access attempts (30-day watch period)

8. **Roles and Responsibilities** — From RACI matrix:
   - CEO: Accountable for all HR security decisions. Approves hiring, termination, NDA.
   - CTO: Responsible for access provisioning/revocation, security awareness technical content.
   - IT Pentester: Assists with access review, security awareness delivery.
   - CISO Agent: Recommends security awareness content, monitors compliance metrics.

9. **Annex A Control Mapping** — Table: A.6.1 (Screening), A.6.2 (Terms and conditions of employment), A.6.3 (Information security awareness, education and training), A.6.4 (Disciplinary process), A.6.5 (Responsibilities after termination or change of employment), A.6.6 (Confidentiality or non-disclosure agreements), A.6.7 (Remote working), A.6.8 (Information security event reporting). Map each to specific sections.

10. **Regulatory Obligations** — ISO 27001 (A.6.1-8), TISAX (ISA Chapter 2), GDPR (employee data handling), NIS2 (informational).

11. **Document Control** — Version history and approval.

**CRITICAL:** Screening procedures must be jurisdiction-appropriate. German and Dutch privacy law limits what employers can check. Keep screening proportionate and legal.
